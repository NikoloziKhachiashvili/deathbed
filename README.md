---

**deathbed** analyses every tracked source file in a git repository and gives it a **health score** based on six real, local metrics ‚ÄĒ no external API calls, no secrets needed.  It then surfaces the files most likely to cause you pain, explains *why* they are dying, and tells you exactly what to do first.

---

## Why?

Every codebase accumulates rot.  Files that nobody owns.  Files too complex to understand.  Files last touched three years ago by someone who left.  These files never show up in sprint planning, but they quietly cause the most bugs, the slowest onboarding, and the worst incidents.

deathbed makes the invisible visible.

## Usage

```bash
# Analyse the current git repo
deathbed

# Analyse a different repo
deathbed --path /path/to/repo

# Show only the 20 worst files
deathbed --top 20

# Show only WARNING and CRITICAL files (score < 65)
deathbed --min-score 65

# Output JSON for CI pipelines / scripting
deathbed --format json

# Combine flags
deathbed --path ~/projects/myapp --top 10 --format json
```

### Options

| Flag | Default | Description |
|------|---------|-------------|
| `--path`, `-p` | `.` | Path to the git repository |
| `--top`, `-t` | `50` | Show only the N worst files (0 = all) |
| `--min-score` | ‚ÄĒ | Only show files with a health score below this value |
| `--format`, `-f` | `rich` | Output format: `rich` or `json` |
| `--version`, `-V` | ‚ÄĒ | Show version and exit |

## Metrics explained

Each file receives a **composite health score from 0‚Äď100** (higher is healthier), built from six weighted sub-scores:

| # | Metric | Weight | What it measures |
|---|--------|--------|-----------------|
| 1 | **Size** | 15% | Lines of code ‚ÄĒ penalises files > 300 / 600 / 1000 lines |
| 2 | **Age** | 20% | Days since any commit touched this file ‚ÄĒ flags abandoned code |
| 3 | **Churn** | 20% | Number of commits to this file ‚ÄĒ instability signal |
| 4 | **Complexity** | 20% | Radon cyclomatic complexity average ‚ÄĒ Python files only; N/A otherwise |
| 5 | **Authors** | 15% | Unique git authors ‚ÄĒ many authors = diffused ownership |
| 6 | **Test coverage** | 10% | Whether a corresponding test file exists anywhere in the repo |

### Health thresholds

| Score | Status | Meaning |
|-------|--------|---------|
| 86‚Äď100 | ‚úÖ HEALTHY | All good |
| 66‚Äď85 | ūüĆ° FAIR | Minor issues |
| 41‚Äď65 | ‚ö†ÔłŹ WARNING | Needs attention soon |
| 0‚Äď40 | ūüíÄ CRITICAL | Actively dangerous |

### Diagnoses

deathbed automatically picks the most meaningful single-phrase diagnosis:

| Diagnosis | What it means |
|-----------|--------------|
| `complexity graveyard` | Cyclomatic complexity is extremely high |
| `legacy ghost` | Not touched in years ‚ÄĒ likely orphaned |
| `too many cooks` | Many authors, nobody owns it |
| `churn monster` | Modified constantly ‚ÄĒ unstable abstraction |
| `growing out of control` | Large and getting larger |
| `nobody's watching this` | Old code with no test coverage |
| `abandoned and complex` | Old *and* hard to understand |
| `healthy` | Nothing to worry about |
